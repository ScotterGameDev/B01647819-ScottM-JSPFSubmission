Lava Hopper

Lava Hopper is a game where the player needs to navigate their way through a difficult pathway in order to get to the other side and escape the lava!
I have created Lava hopper using many different techniques and will touch on some of them here


Building The Level

Lava

When it came to building the level, I began with a terrain built from a height map. This allowed me to encase the player in the level and have the
play area be obvious to the player. I done this by creating a height map which gave an indented look to the level, allowing me to then create
'grounds'. These grounds consisted of one ground which would be the rotating lava which would not detect collisions and then another
(which would be slightly below) which would gather collisions. I done this as when I added collisions to the rotating ground, it would stop rotating.




Lights

Next up was the level lighting. As you can see above, the level had the look I was going for but it was missing something. I decided to add some semi-intense
red lighting which really helped the level and lava look much more menacing. To begin with, the first function is creating a hemispheric light that will
light the whole scene so the level is at least visible to begin with. Next I created the main red spotlight that would cast over the whole level, making it
seem as if the red light was coming from the lava, using diffuse and specular to ensure it spreads out and affects other lights too. Thirdly, I added a miniature
spotlight over the end level goal, to try and make it stand out to the player as a place they need to reach.





Player Character

The player character was mainly chosen to resemble a sort of test dummy look. Mainly because this is the exact kind of position that only a test dummy
would find themselves in. Using this model, I was able to attach animations and also look for collisions with other objects too! Adding these things
brought much more life to the game as well as handled the main win and loss conditions. Firstly, the movement; By checking for which key has been
pressed, the player character will either move forward by using the "moveWithCollisions" method, backwards by using the same method but with 'minus' speed
or rotate left and right by checking the button and then rotating in the correct direction at the set rotation speed.


Secondly, the collisions were also handled inside the players movement script. This was done as the game is handled through the player colliding with
and interacting with certain objects to both win or lose. If the player collides with the lava, then the game is lost, however if the player collides
with the end goal object, then they win the game! So, this code will check if the player is interacting with the platforms and return a console message
to ensure the player is in fact on the platforms and not hitting the lava. Then, the collision is checking if the player has collided with the
level objective floating box, if so, reset the players position back to the beginning and reset the rotation too. Then, create an audio method to play
the winning sound, then play it and set the scene index to load the 'game won' scene. This is then done again but for if the player has fallen off the
level and hit the lava, the player will be reset, although no noise will play and the player will be sent to the 'you lost' scene.


Thirdly, the players animations. This script will set a boolean for if the object is currently animating or not. This is done by checking if the player is
moving and has already started animating. If so, the code will only be called once, whereas if we didn't check if the player was animating, the code would
continue to be called and would result in major frame rate issues as the game went on. This block of code also checks if the player
is still playing the walking animation but is not moving. In which case, play the idle animation and reset the original animation trigger boolean. Doing so
so the whole thing can be re-triggered again when the player starts moving again.



Main Menu

Main Menu Button


Lastly, I created a button to allow the player to go from the main menu to the game. This was coloured to suit the red themes going through the geame
already and was also done to stand out from the very dark background. To create the button I used the GUI method to create a simple button. I created
parameters to decide where the button should be placed in the scene, how wide and tall it should be, the button colour, how smoothed out the edges are and
then the buttons background colour. Below that I created two sounds to play when the button is pressed, one to register the button click and another to play
throughout the level, making sure that one is set to loop. Then, when the button is pressed, I play the two sounds and change the scene.
This was also used to create the 'Return To Menu' buttons on both of the 'Game Won' and 'Game Lost' screens.
